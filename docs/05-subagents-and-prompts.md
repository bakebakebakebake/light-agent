# 05 · 子 Agent 与提示词(Sub-agents & Prompts)

> 原创整理,基于公开信息与通用 agent 设计模式。代码为示意性伪代码。
> 注意:本章不包含任何真实 system prompt 的逐字内容——那属于专有材料。这里只讲**结构与设计意图**。

## 子 Agent:用隔离上下文防止"上下文爆炸"

当一个任务需要大量探索(读很多文件、跑很多搜索)时,如果都塞进主对话,主上下文很快会被冗长的中间过程撑爆。

解法:**派生子 agent 在隔离的上下文里干活,只把摘要返回给父 agent。**

```
父 agent
  │  "去调研 X,把关键发现总结给我"
  ▼
子 agent(独立上下文)
  - 读 20 个文件、跑 10 次搜索……(这些都不进父上下文)
  - 归纳出结论
  │  只返回:简短摘要
  ▼
父 agent(上下文只增加了那段摘要)
```

收益:
- **保护父上下文**:冗长的中间工作留在子 agent 里。
- **可并行**:多个独立子任务可并发跑。

## Skills vs Sub-agents:成本/隔离权衡

两者容易混,但取舍点很清楚:

| | Skill | Sub-agent |
|---|---|---|
| 注入方式 | 注入**当前**上下文 | 创建**新的隔离**上下文 |
| 成本 | 低 | 高(独立上下文开销) |
| 适用 | 轻量、就地增强能力 | 重活、需要隔离防爆炸 |

一句话:**Skill 便宜但共享上下文;spawn 一个 agent 贵但隔离,能防止上下文爆炸。** 按任务体量选。

另一条约束(与安全章呼应):**用户的显式权限决定,永远覆盖子 agent 自己的权限模式。**

## 何时用子 agent / 何时不用

- **该用**:任务在多个独立项上"扇出"(很多文件要读、很多候选要查、很多测试要跑);开放式深度调研。
- **不该用**:目标已经明确、单点查找(读某个具体文件、grep 某个符号)——直接用工具更快,别为了用而用。

## System Prompt:用提示词驱动行为,而非硬编码

一个重要的架构取向:**把行为规范、工具使用约定、安全规则写进 system prompt,而不是硬编进程序逻辑。**

好处:
- **可调**:改行为改提示词即可,不必改代码。
- **可审计**:行为规范集中、可读。
- **灵活**:同一套循环 + 不同提示词 = 不同人格/专长的 agent。

### System Prompt 的常见结构(组织方式,非逐字内容)

一个组织良好的 system prompt 通常分这几块:

```
1. 身份与角色      —— 你是谁、为谁服务
2. 能力与边界      —— 能做什么、不能做什么
3. 工具使用约定    —— 何时用哪个工具、并行规则、首选专用工具
4. 安全与权限规则  —— deny-first、高风险先确认、injection 防御
5. 风格指南        —— 输出简洁度、格式、语气
6. 任务方法论      —— 如何规划、验证、收尾
7. 环境上下文      —— 工作目录、平台、日期等运行时信息
```

> ⚠️ 这里只给**结构骨架**。具体每条该怎么写,应由你根据自己 agent 的定位原创撰写,不要去复制任何专有提示词的原文。

## Prompt Cache:提示词稳定性即成本

提示词的**稳定性**直接关系 prompt cache 命中率(详见 [03 · 上下文管理](03-context-management.md)):

- system prompt、工具 schema、模型、headers 任一变动都可能让缓存失效。
- 因此:**会话开始时把 system prompt 与工具 schema 定下来,中途尽量别改。**
- feature flag 用 **sticky latch**:中途切模式不改 system prompt,从而不破坏缓存。

## 设计要点清单

- [ ] 重活/调研派给子 agent,隔离上下文,只回传摘要。
- [ ] 分清 Skill(便宜、共享上下文)与 sub-agent(贵、隔离)。
- [ ] 明确目标的单点查找直接用工具,别滥用子 agent。
- [ ] 行为/工具/安全规则写进 system prompt,而非硬编码。
- [ ] system prompt 分块组织;具体内容自己原创。
- [ ] 保持 prompt 稳定以保护缓存;flag 用 sticky latch。
