# 11 · 可观测性、评估与验证(Observability, Eval & Verification)

> 原创整理,基于公开信息与通用 agent 设计模式。代码为示意性伪代码。

agent 是个不确定系统,**看不见就改不动**。这一章讲三件让 agent 可生产化的事:看清它在干嘛(可观测性)、判断它好不好(评估)、保证它没干坏(验证)。

## 一、可观测性(Observability)

### 结构化日志 / 遥测

把每一步关键事件记成**结构化**记录(配合 [07](07-session-lifecycle.md) 的 transcript):

```
事件:model_call / tool_call / tool_result / compaction / permission_prompt / error
每条带:session_id、轮次、时间戳、耗时、token 用量、结果状态
```

- **结构化**(JSON)而非纯文本,便于检索聚合。
- 关联 `session_id` + 轮次,能把一次完整任务串起来回放。

### 成本追踪(Cost Tracking)

token 直接对应钱和延迟,必须可见(呼应 [06](06-model-interaction.md) token 预算):

- 记录每轮 **input / output / cache 命中** token。
- 按会话、按模型、按工具聚合成本。
- **缓存命中率**是关键指标——缓存优化(见 [03](03-context-management.md))的效果要靠它衡量。

### 调试支持

- transcript 可**回放**,定位"它当时为什么这么决策"。
- 工具调用的输入输出完整留痕(大输出落盘 + 引用,见 [03](03-context-management.md))。
- 错误带上下文:哪一步、什么输入、什么 stop_reason。

## 二、评估(Evaluation)

agent 行为不确定,**改一处可能悄悄弄坏另一处**,所以需要系统化评估而非靠手感。

### 评估数据集

- 构建一组**代表性任务**(覆盖典型场景 + 已知难例 + 历史 bug 复现)。
- 每个任务定义**成功判据**:产物是否正确、是否通过测试、是否避开危险操作。

### 评估维度

| 维度 | 看什么 |
|---|---|
| 正确性 | 任务是否真正完成、结果对不对 |
| 效率 | 用了多少轮、多少 token、多少时间 |
| 安全 | 是否恰当地触发确认、是否避开危险操作 |
| 鲁棒性 | 对模糊/异常输入的处理 |

### 评判方式

- **确定性检查优先**:能用"跑测试通过没""文件 diff 对不对"判定的,就别用主观判断。
- **LLM-as-judge**:难以程序化判定的(回答质量、推理合理性),用模型按 rubric 打分;注意它本身有偏差,需校准。
- **回归测试**:把修过的 bug 沉淀成 eval case,防止复发(尤其是 prompt 改动后)。

### 注意 prompt 改动的回归

改 system prompt(见 [05](05-subagents-and-prompts.md))是高风险动作——可能在你没注意的地方改变行为。**每次改 prompt 都应跑一遍 eval**。

## 三、验证(Verification)——agent 自己对自己的检查

这是 agent **在交付前自查**的能力,直接提升可靠性。

### 改完就验证

任何代码改动后,agent 应**主动跑项目的 build / test**,而不是改完就宣布完成:

```
做出改动 → 跑 build/compile → 跑相关测试 → 有错先修再交付
```

- 先**探明项目用什么工具链**(看 package.json / Makefile / pom.xml 等),用对的命令。
- 新增功能/修 bug 时**写测试并跑**;没有测试框架就按生态惯例搭一个。
- 跑不动(缺依赖/环境受限)就**如实说明**,而不是假装通过。

### 失败循环的识别

一个重要的自我约束:**同一方法失败两次,就停下来诊断根因,而不是继续打补丁**。换一个根本不同的思路,必要时向用户说明偏差再继续(呼应总体设计哲学)。

### 安全敏感改动要说清验证边界

对 auth / 基础设施 / 数据处理类改动,明确**验证了什么、没能验证什么**,不要把"假设"当"事实"陈述。

## 把三者串起来

```
可观测性  →  提供数据(发生了什么、花了多少)
   ↓
评估      →  基于数据判断好坏、防回归
   ↓
验证      →  在每次交付前把住质量关
```

三者共享 [07](07-session-lifecycle.md) 的 transcript 作为数据底座。

## 设计要点清单

- [ ] 结构化日志,关联 session_id+轮次,可回放。
- [ ] 追踪每轮 token 与成本,盯紧缓存命中率。
- [ ] 错误带足够上下文,便于定位。
- [ ] 建代表性 eval 数据集 + 明确成功判据。
- [ ] 评估覆盖正确性/效率/安全/鲁棒性。
- [ ] 确定性检查优先,LLM-as-judge 补充,bug 沉淀为回归用例。
- [ ] 每次改 prompt 都跑 eval。
- [ ] 改完主动跑 build/test,先修再交付。
- [ ] 同一方法失败两次就诊断根因、换思路。
- [ ] 安全敏感改动说清验证边界。
