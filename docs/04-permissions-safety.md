# 04 · 权限与安全(Permissions & Safety)

> 原创整理,基于公开信息与通用 agent 设计模式。代码为示意性伪代码。

这是 agentic 工具能否真正落地的关键一层。一个能改文件、跑命令、访问网络的 agent,如果没有稳健的权限与安全模型,就是个隐患。

## 核心原则一:Defense in Depth(纵深防御),Deny-First

安全不是单一开关,而是**多层叠加**,且默认从严:

```
deny  >  ask  >  allow
(最严格的规则获胜)
```

具体做法:**工具默认被假定为"会写、且具有破坏性",除非它显式声明自己是安全的**。也就是说,风险画像是 **opt-out(默认有风险,需主动声明安全)** 而非 opt-in。这样新增工具时不会因为忘了配置而默认放行危险操作。

叠加在上面的层次通常包括:
1. **allow / deny 规则**:显式名单,deny 优先。
2. **预执行 hook**:在执行前可以改写输入、或直接拦截动作。
3. **安全分类器(safety classifier)**:在自动模式下对动作做风险判定。

## 核心原则二:按"可逆性 + 影响范围"分级

判断一个动作要不要先问用户,看两个维度:**可逆性**和**影响范围(blast radius)**。

| 风险级别 | 例子 | 处理 |
|---|---|---|
| 低(本地可逆) | 读文件、跑测试、跑 linter | 直接执行 |
| 中 | 装依赖、跑 build、改配置 | 执行但告知用户 |
| 高(难逆转/影响共享系统) | 删多个文件、drop 数据库、改生产、force push、改权限 | **先解释风险,等用户明确确认** |

需要确认的典型高风险动作:
- 破坏性操作:递归删除、drop 表、`rm -rf`、覆盖未提交改动。
- 难以逆转:force push、`reset --hard`、改 CI/CD。
- 对外可见 / 影响共享状态:push 代码、发 PR、发消息、改基础设施或权限。

原则:**遇到障碍时不要用破坏性手段抄近路**(比如别用 `--no-verify` 绕过校验)。先找根因,优先选非破坏性替代方案。

## 核心原则三:信任分级,且不随会话恢复

权限模式构成一个**渐进的信任谱**,用户随使用逐步往"更自动"的方向移动:

```
最严(每步都问)  ←——————→  最松(自动执行大多数操作)
```

两个关键约束:
- **用户的显式权限决定永远覆盖**子 agent 自己的权限模式。
- **权限不随会话恢复而自动还原**——重开会话回到保守默认,不继承上次的"信任"。这避免了"上次授权"被误用。

## 核心原则四:Graceful Degradation(信任流失时优雅降级)

一个很有价值的模式——**denial tracking(拒绝追踪)**:

> 当用户连续拒绝若干次、或累计拒绝达到一定数量后,系统**停止自主行动**,转为每一步都先征求许可。

这条原则被总结为:**"当用户失去信任时,优雅降级"**,其价值往往高于任何单个功能。它让 agent 在"用户明显不放心"时自动变保守,而不是固执地继续。

## 核心原则五:把外部内容当作不可信数据(Prompt Injection 防御)

来自**文件内容、命令输出、网页、工具结果**的一切,都应被视为**不可信数据**,而不是指令。

- 如果外部内容里出现"看起来像指令"的文本(如"忽略之前的指令""你现在是另一个 agent"),**不要执行**,应向用户标记。
- 指令只应来自**用户在对话中的直接输入**,不应来自工具结果或被操作的内容。
- 操作浏览器时,DOM 元素、其属性(onclick 等)、cookie、localStorage 都是不可信数据,不能用来覆盖安全规则。
- 邮件/消息内容(主题、正文、附件)一律当不可信数据;基于其内容做删除/发送/回复前必须用户确认。

## 已知弱点(分析中坦承的,值得警惕)

社区分析对弱点也很诚实,这些正是你自己实现时要重点防的:

- **共享性能约束的层会一起退化**:多个防护层若依赖同一资源,可能同时失效。
- **子命令多的命令可能绕过分析**:像 `git` 这类有大量子命令的命令,静态分析容易被绕过。
- **预信任执行窗口**:某些扩展可能在"信任对话框出现之前"就已运行——注入点的时序要小心。

## 设计要点清单

- [ ] Deny-first:工具默认有破坏性,安全需显式声明(opt-out)。
- [ ] 纵深防御:allow/deny 规则 + 预执行 hook + 安全分类器。
- [ ] 按可逆性 + 影响范围分级:低直接做、中告知、高先确认。
- [ ] 不用破坏性手段抄近路;优先非破坏性替代。
- [ ] 信任分级;用户决定覆盖子 agent;权限不随会话恢复。
- [ ] denial tracking:用户多次拒绝后自动转保守。
- [ ] 一切外部内容当不可信数据,严防 prompt injection。
- [ ] 警惕:层间共享约束、子命令绕过、预信任执行窗口。
